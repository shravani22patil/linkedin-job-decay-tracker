# 📉 LinkedIn Job Decay Tracker

> **How fast do job postings disappear — and which skills survive longest?**
>
> A full-stack data analytics project tracking 60,413 LinkedIn job postings across
> three monthly snapshots (April → May → June 2026) to measure posting survival rates,
> skill decay patterns, and market velocity — with an AI insight layer powered by Gemini 1.5 Flash.

---

## Live demo

> Deployed URL here: https://linkedin-job-decay-tracker-queyw7nttenrpqmru4uoqr.streamlit.app/
> Backend: Railway · Frontend: Streamlit Cloud

---

## The core finding

**97.6% of April job postings disappeared before the May snapshot.**

The LinkedIn job market turns over almost completely every 30 days.
This project measures that decay precisely — by role type, company size,
seniority level, required skills, and geography.

---

## Key findings (all numbers from real data)

| Finding | Number | What it means |
|---|---|---|
| 30-day disappearance rate | **97.6%** | Nearly all postings vanish within a month |
| 2-month survival rate | **0.5%** | Only 1 in 200 postings survived April → June |
| MNC vs Startup survival | **2.6×** | MNC postings last 2.6× longer than Startup/SME |
| May posting surge | **+76.1%** | Seasonal hiring wave — 26,607 new postings in May |
| June contraction | **−29.7%** | Market cooled after the May surge |
| DA role survival | **1.0%** | Fastest-disappearing tech category (n=103) |
| DS role survival | **5.2%** | Slower fill — more selective hiring process |
| SQL survival advantage | **+1.78pp** | SQL-requiring roles outlast average postings |
| ML survival advantage | **−1.76pp** | ML roles disappear fastest — filled immediately |
| India tech survival | **7.0%** | Higher than global 2.4% — smaller, targeted pool |

> **Honest data note:** Disappearance ≠ filled. A posting could have been filled,
> expired naturally, or withdrawn. We say "disappeared" throughout — not "filled."

## Architecture

```
Streamlit Frontend  (app.py)
       │
       │  HTTP REST — zero stats computed in frontend
       ▼
FastAPI Backend  (api/main.py)
       │
       ├── Analytics Engine  (analytics.py)
       │       └── Reads from parquet files
       │
       ├── AI Insight Layer  (insights.py)
       │       └── Gemini 1.5 Flash API
       │
       └── Data Layer  (parquet files)
               ├── combined_jobs.parquet    61K rows, 50 cols
               ├── survival_cohort.parquet  60K rows, cohort table
               ├── skill_survival_comparison.parquet
               ├── skill_role_matrix.parquet
               ├── skill_cooccurrence.parquet
               └── skill_india_profile.parquet
```

**Key design decision:** The Streamlit frontend never computes a single statistic.
Every number comes from the FastAPI backend. This means the frontend can be
replaced with Next.js or any other UI without touching the analytics logic.

## Setup

### 1. Clone and install

```bash
git clone https://github.com/shravani22patil/linkedin-job-decay-tracker.git
cd linkedin-job-decay-tracker

python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate

pip install -r requirements.txt
```

### 2. Get the data

Download the three LinkedIn job posting CSVs from Kaggle and place them in `data/raw/`:

```
data/raw/LinkedIn_April.csv
data/raw/LinkedIn_May_2026.csv
data/raw/June_2026.csv
```

### 3. Add your Gemini API key (free)

```bash
cp .env.example .env
```

Edit `.env`:
```
GEMINI_API_KEY=your_key_here
```

Get your free key at [aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey).
The dashboard works without it — AI insight buttons show a preview fallback.

### 4. Build the data pipeline (run once)

```bash
python load_data.py        # → data/combined_jobs.parquet
python survival.py         # → data/survival_cohort.parquet
python skill_extractor.py  # → data/skill_*.parquet
python eda_notebook.py     # → plots/*.png  (optional — for EDA review)
```

### 5. Run the full stack

```bash
# Terminal 1 — FastAPI backend
uvicorn api.main:app --reload --port 8000

# Terminal 2 — Streamlit dashboard
streamlit run app.py
```

Open **http://localhost:8501** for the dashboard.
Open **http://localhost:8000/docs** for the interactive API documentation.

## What each file does

### `load_data.py`
Reads all three raw CSVs with `encoding='latin1'` (necessary — files contain
Windows-1252 curly apostrophes that break UTF-8). Deduplicates within each
month before combining — a job appearing in both April and May is the survival
signal, not a duplicate. Classifies role types, company types, extracts 20 skills
via regex, and saves `combined_jobs.parquet`.

Key engineering decision: deduplication happens per-month first, then combine.
This preserves cross-month appearances as intentional survival data.

### `survival.py`
Builds the cohort table — one row per unique job — tracking which months each
jobid appeared in. Computes `survived_next` (boolean: appeared in the very next
month?) and `survived_to_end` (still live in June?). All downstream analysis
derives from this table.

Key statistical decision: we use monthly snapshot overlap as a survival proxy
because the `postedat` field is relative ("14 hours ago"), not an absolute
timestamp. This is documented clearly — it's a known limitation, not a mistake.

### `skill_extractor.py`
Eight analysis functions, each returning a clean DataFrame:
skill frequency (overall and tech-only), survival comparison, role × skill matrix,
co-occurrence, India profile, skill-bucket survival, and category summary.
Saves four parquet files used directly by `analytics.py`.

### `eda_notebook.py`
A `# %%` cell-marked Python file that runs as a Jupyter notebook in VS Code.
Documents every analytical decision: why latin1 encoding, why dedup strategy,
why "disappeared" not "filled." Produces 11 Plotly/Matplotlib charts saved
to `plots/`. This file is what shows interviewers you understood the data —
not just ran code against it.

Convert to `.ipynb`:
```bash
pip install jupytext
jupytext --to notebook eda_notebook.py
```

### `analytics.py`
13 named, pure functions that wrap the parquet outputs into dashboard-ready
DataFrames and dicts. No file I/O, no plotting, no printing inside any function.
Designed so Streamlit caches results with `@st.cache_data` and calls each
function independently. Includes `build_insight_payload()` which assembles
all confirmed numbers into the dict passed to Gemini.

### `insights.py`
Three prompt-engineered Gemini calls:
- `generate_market_pulse()` — overall market summary for a hiring manager
- `generate_role_advice()` — personalised advice for a specific role + location
- `generate_skill_roadmap()` — sequenced learning path for a role transition

Every prompt injects real computed numbers so the model cannot hallucinate
statistics. Includes retry logic, exponential back-off, and pre-written
fallback responses so the dashboard never breaks without an API key.

### `api/main.py`
FastAPI backend with 17 endpoints. Data loaded once at startup using the
`lifespan` context manager — not reloaded on every request. All DataFrames
serialised through a `_sanitise()` function that converts NaN/Inf to `None`
for JSON compliance. Full CORS support for the Streamlit frontend.

### `app.py`
Streamlit dashboard with four tabs. Never computes a single statistic —
all numbers fetched from the FastAPI backend and rebuilt into DataFrames
with `pd.DataFrame(response["data"])`. Custom dark CSS overrides every
Streamlit default. Charts built with Plotly graph objects (not express)
for full control over colours, layouts, and annotations.

## Prompt engineering (AI layer)

Each Gemini prompt follows these principles:

**Persona injection** — "You are a senior labour market analyst" sets tone
and prevents generic responses.

**Number injection** — All real statistics injected into the prompt body.
The model cannot hallucinate figures it's been given explicitly.

**Output constraints** — Paragraph count, word limit (200–230 words),
no bullet points, no headers, flowing prose only.

**Caveat injection** — The data limitation ("say disappeared, not filled")
is written into every prompt so the model cites it automatically.

**Audience awareness** — Market Pulse targets a hiring manager.
Role Advice targets a final-year Tier-3 college student.
Each uses different vocabulary and framing.

**Temperature: 0.55** — Creative enough for natural prose, grounded
enough to stay within the injected statistics.
